# WBS ツール - CLAUDE.md

テキスト（JSON）駆動の WBS／ガント＋イナズマ線ツール。
`wbs.json`（データ）を single source of truth とし、JS内蔵の単一HTML `wbs_viewer.html` が描画する。

---

## 取り扱いの原則（重要）

- **`wbs_viewer.html`（ビューア）は原則さわらない。** 表示ロジックは完成済み。
- **変更は `wbs.json`（データ）の編集のみ。** AI に作業を頼むときも、原則 JSON を直す。
  - 表示仕様そのものを変えたい場合に限り HTML を触る（その時は明示的に依頼）。

## アーカイブ運用（過去タスクの整理）

項目が増えすぎないよう、**古い完了タスクは current から退避**する：

- **トリガー**：ガントの表示が進み、**過去の月（例：2026年5月以前）の完了タスク**が
  「過去日付以前」になったら実施。
- **手順**：
  1. **同じフォルダ内にバックアップを作成**（例：`wbs-archive-2026-05.json` に退避分をまとめる）。
  2. `wbs.json` から該当の**実施済み（完了）タスクを削除**する。
- → current の `wbs.json` は「今と未来」に集中。履歴はアーカイブ JSON に保全。
- 大量項目対策は **折りたたみ（表示）＋このアーカイブ運用（データ）** の二段で行う。

---

## 動作環境
- **Google Chrome（最新版）専用**。HTTPサーバー不要・外部ライブラリ/CDN不要。
- `wbs_viewer.html` を **file:// でそのまま開く**。

## 開き方・更新
1. Chrome で `wbs_viewer.html` を開く。
2. **「ファイルを開く」** で `wbs.json` を選ぶ（または操作バーへ **D&D**）。
3. `wbs.json` を編集して保存 → **「更新」ボタン**で再読込・再描画（再選択不要 / File System Access API）。
4. **全展開 / 全たたみ** ボタン、または **◆プロジェクト・1L・2L の `▼/▶`〜名前をクリック**で**折りたたみ**。
5. スクロール位置は**折りたたみ・更新では保持**、**新規ファイル読込時のみ本日へセンタリング＋折りたたみリセット**。

## ブラウザ編集（編集モード）
テキスト/AI 編集に加えて、**画面上から直接 `wbs.json` を編集**できる（任意機能）。
- **「編集」ボタン**でON/OFF（ONで緑）。ONにすると左の表が入力欄＋操作ボタンに変わる。
- できること：**各フィールドのインライン編集**（No.=id／作業項目名／数量／時間／担当／予定・実績の日付＝ピッカー／備考）、
  **リーフ追加**（行の `＋`＝下に兄弟、プロジェクト行の `＋タスク`。id は既存と重複しないよう採番。折りたたみ中の追加は自動展開）、
  **削除**（`✕`・確認あり、配下ごと）、**兄弟内の上下移動**（`▲▼`）。
- **自動保存**：変更は約0.4秒のデバウンス後に `wbs.json` へ書き戻し（File System Access API）。
  書込は単一キューで直列化。除外するのは内部派生値 `_calc`/`_leaf` のみ（**`_` 始まりのユーザーキーは保持**）。
- **外部変更の検知**：保存直前にファイルの更新時刻を確認し、ツール外（AI編集等）で変わっていたら**上書き確認**を出す。
  AI に編集を頼む時は編集モードをOFFにしておくのが安全。
- 前提：保存には**書き込み権限**が要る。「ファイルを開く」or D&D で開いた**ハンドルがあるファイルのみ**編集可。
  **file:// では書き込み許可のプロンプトが出せない**ため、編集ON時に**保存ダイアログで同じ wbs.json を選び直す**（選択自体が許可になる）。
  「もう一度押してください」と出たら**再度「編集」を押す**（ブラウザのジェスチャー制限。2回目は直接ダイアログが開く）。
  この許可は**セッション限り**（Chrome を再起動したら、編集ONのたびに再選択が必要）。
  **新規ファイルを読み込むと編集モードは自動OFF**（再度ONで新ファイルの権限を取得）。読み込み失敗時はハンドルを差し替えない（別ファイルを誤上書きしない）。
- **旧フォーマット**（`{project,tasks}`）は編集ON時に確認の上、**新フォーマット（`projects[]`）へ変換**してから編集する。
- 入力ガード：日付は **1900〜2099年のみ有効**（範囲外・入力途中の値は無視して保存しない）。数量/時間は小数可（`step="any"`）。
  number 入力上のホイールは無効（誤変更防止）。タブを閉じる際、未保存があれば確認が出る。
- 範囲外（＝やらないこと）：ドラッグ&ドロップ／親をまたぐ移動／自動リナンバー／リーフ↔集計の変換／プロジェクト追加。これらは JSON か AI 編集で。
- 工数・進捗・イナズマ線は従来どおり自動再計算（編集中は入力フォーカスが外れてから再描画）。

## データ（wbs.json）
- **複数プロジェクト対応**。トップは `projects: [{ name, milestones, tasks }]`。
  - 旧フォーマット（`{ project, milestones, tasks }` 単一）も**後方互換で読める**。
- 各プロジェクトの `tasks` は親子ネスト（最大3階層）。`children` あり＝集計、なし＝リーフ（工数を持つ）。
- リーフのフィールド：
  ```
  id, name, qty(数量), hours(1単位あたり時間), assignee(担当),
  plan:   { start, end }        予定 "YYYY-MM-DD"
  actual: { start, end }        実績（未定は null）
  note
  ```
- **工数・進捗率はデータに持たせない**（すべて派生値。HTML側のJSが算出）。
- `milestones: [{ date, label, color }]` は各プロジェクト内に任意。

## データの追加・更新（やり方）

編集は `wbs.json` のみ。保存 → 画面の **「更新」** ボタンで反映。

### ① 状態の更新（日々の運用・最重要）
進捗率・工数・イナズマ線は**自動計算**。手で触るのは **「実績の日付」だけ**：
- **着手したら** → そのリーフの `actual.start` に日付（→ 実績終了列が「進行中」・イナズマ線に反映）
- **完了したら** → `actual.end` に日付（→ 行がグレー＋`✓`・イナズマ点は本日線）
- **予定変更** → `plan.start` / `plan.end` を直す

### ② 作業項目（タスク）を追加
集計ノードの `children` にリーフを1件足す：
```json
{ "id": "2.1.3", "name": "新規作業", "qty": 1, "hours": 16, "assignee": "ぴぐお",
  "plan":   { "start": "2026-07-01", "end": "2026-07-05" },
  "actual": { "start": null, "end": null }, "note": "" }
```
- 中間の工程を増やすなら、`children` を持つ**集計ノード**を足してその下にリーフを入れる（最大3階層）。

### ③ プロジェクトを追加
`projects` 配列に1件足す：
```json
{ "name": "新プロジェクト", "milestones": [],
  "tasks": [ { "id": "1", "name": "フェーズ1", "children": [ /* リーフ… */ ] } ] }
```

### ④ マイルストーンを追加
該当プロジェクトの `milestones` に：
```json
{ "date": "2026-09-30", "label": "リリース", "color": "#ef4444" }
```

### ⑤ AI への依頼例（原則 JSON を編集させる）
- 「設計レビューを**今日完了**にして」→ 該当リーフの `actual.end` に本日を入れる
- 「コンポーネント配置に**着手**」→ `actual.start` を入れる
- 「テストフェーズを**追加**して」→ 集計ノード＋リーフを追記
- 「5月以前の完了を**アーカイブ**して」→ 下記アーカイブ運用（バックアップ＋削除）

## 計算ロジック（HTML側で決定論的に算出）
- **工数（人日）= `qty × hours ÷ 8`**。親 = 配下リーフの合計。
- 親の予定/実績期間 = 配下リーフの最小start〜最大end。
- 進捗率（基準日＝本日）：
  - 未着手（actual.start なし）= 0% ／ 完了（actual.end あり）= 100%
  - 進行中 = `clamp((本日 − actual.start) ÷ (plan.end − plan.start) × 100, 0, 100)`
  - 親 = 配下リーフの (progress × 工数) の加重平均。
- ヘッダに「プロジェクト数 ／ 総工数(人日)」。
- 進捗率は内部で算出し**イナズマ線・親集計に使うのみ**（時間ベースで誤解を招くため、数値としては表示しない）。

## 表示
- 列見出しは中央寄せ。左の情報表（No.〜備考）は固定、**横スクロールはガントだけ**。
- **予定/実績の日付列は2段見出し**：上段に「予定」「実績」、その下に「開始／終了」（`COLS` の `group` プロパティで隣接列をまとめる）。
- 行のレイヤ色：**◆プロジェクト（上下に区切り線）＞ 1L ＞ 2L ＞ 3L**。
- ガント：日付＋曜日（年月は年あり）、土日グレー、上段=予定（淡）/下段=実績（濃）、進行中は実績を本日まで。
- 日付列（予定/実績）は `5/11` 表示。初期表示は本日が中央付近。
- **完了タスク**：行を濃いめグレー＋取り消し線＋先頭 `✓`。
- オーバーレイ（SVG）：
  - **イナズマ線**（赤・**全プロジェクト通しで1本**）。各 terminal 行の点を次で決める（**左へ突出＝遅延**）：
    - 完了 ＝ 本日線上
    - **終了遅延**（着手済み・未完了で本日 > `plan.end`）＝ **予定終了日**（締切超過を優先表示）
    - **着手遅延**（未着手で本日 > `plan.start`）＝ **予定開始日**
    - 進行中（期限内）＝ 本日線上（着手ズレは表示しない）
    - それ以外（未着手で未到来 等）＝ 本日線上
    - 折りたたんだノードは集計値で1点。本日線は引かない（基準＝本日）。
  - **マイルストーン線**（各プロジェクトの `milestones`・任意色）。

## 異常系（壊れた入力の扱い）

**方針：壊れた入力でもクラッシュさせない（graceful degradation）。** 無効値は無視・0・空で表示する。
データ投入時は下記を避ける（入れても落ちないが表示は崩れる）。

| 入力（異常） | ビューアの扱い |
|---|---|
| 不正な日付（`"YYYY-MM-DD"` 以外・**1900〜2099年の範囲外**）| **無視**（plan/actual/マイルストーン共通）。日付欄は `—` |
| マイルストーンの `label` 欠損 | **空ラベルで描画**（クラッシュしない） |
| 進行中なのに `plan.end` 欠損 | 進捗 **0%**（NaN を回避） |
| 終了 < 開始（期間の逆転） | バーが描画されない／異常な見た目 |
| 実績開始が未来日 | 進捗 0 |
| `qty` / `hours` 欠損・0 | 工数 0（空表示） |
| 少数の `qty`（例 0.5） | 小数の人日（1.5 等） |
| 4 階層を超えるネスト | 色は **3L 止まり**（崩れはしない） |
| 範囲外・色なし・不正日付のマイルストーン | **描画されない**（無視） |
| マイルストーンの `color` が `#hex` 以外 | **無視してデフォルト色**（属性インジェクション防止） |
| 同一プロジェクト内の **重複 id** | 折りたたみキーが衝突（同 id が同時に開閉） |
| `projects: []` ／ `tasks: []`（空） | 空表示（クラッシュしない） |
| トップが `null`・数値等の非オブジェクト | 空表示（クラッシュしない） |
| 不正な JSON（D&D / ファイル選択 / 更新） | アラートで通知（無言失敗しない） |

- これらの確認用データは **`tests/異常_*.json`**、正常系は **`tests/正常_*.json`**（一覧は `tests/INDEX.md`）。
- 「クラッシュしないこと」は headless Chromium で全テストを読み込み、**JS エラー・`NaN` 無し**を確認済み。

## 既知の制限
- **大量行（数千〜）で初期描画が重くなる**（毎回 innerHTML 全再構築）。対策＝折りたたみ＋アーカイブ運用。本当に重くなったら仮想化が必要。
- **同名プロジェクトは折りたたみキーが衝突**（一緒に開閉する）。プロジェクト名は一意に。
- **配列内の `null` 要素（null のタスク/プロジェクト）は非対応**（トップの非オブジェクトは空表示で吸収）。
- **キーボード操作・スクリーンリーダー対応なし**（マウス前提の個人ツール）。
- **編集モードでプロジェクト名や id を変えると折りたたみ状態がリセット**される（折りたたみキーが名前/id 由来のため）。
- **複数行の `note` は編集モードで触ると1行に平坦化**される（input が改行を保持できない）。

## メモ
- Chrome 専用・file:// 前提（File System Access API を使用）。
- 実データを公開したくない場合は `wbs.json` を gitignore してよい（中身は手元のみ）。`wbs-archive-*.json` は既定で無視。
