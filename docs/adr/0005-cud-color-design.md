# ADR-0005: 配色はCUD配慮（色だけに意味を担わせない）

色だけに意味を持たせず、形・位置・ラベルで冗長化する（色覚多様性への配慮）。

## ステータス
承認済み（実装：#34 / #65）。CUDの検証・調整は **#35（OPEN・継続）** で追跡。

## コンテキスト
ガント・進捗バー・イナズマ線・階層・状態（完了/遅延）と、色を使う場面が多い。色覚多様性（CUD）で区別がつかなくなる配色は誤読を生む。MS既定色は「青と紫」が紛れやすい。

## 決定
**Okabe-Ito 原則**に沿い、色だけで意味を伝えず**形・位置・ラベルで冗長化**する。
- 1色＝1意味（実績/予定/遅延/完了の割り当て）。**「青と紫」を避ける**（既定マイルストーン色はモーブ `#cc79a7`）。
- 親は色相でなく**バーの太さ**、着手遅れは色でなく**枠の空き**で区別する。
- 色ごとの割り当ての詳細仕様は [`CLAUDE.md`「配色（CUD配慮・#35）」](../../CLAUDE.md)（重複させない）。

## 理由
- 1色＝1意味を守ると、色数が増えても各色の情報量が落ちない。
- 形・位置・ラベルの冗長化で、色覚3型（P/D/T）でも区別が残る。
- ガント刷新（#65）で「予定=枠／実績=塗り」にしたのも、プログレスバー誤認を形で防ぐため。

## 結果
- 配色を**足す/変える変更では配色監査を必須化**：`tests/e2e/test_color_audit.py` が重なる/隣接するペアの ΔE を色覚3型で評価し、ΔE<10 の新規ペアをゼロに保つ（冗長化済みなら理由つきで `ACCEPTED`）。
- 監査の閾値根拠（Machado 2009 のCVD行列・ΔE76・WCAG輝度比）はスクリプト冒頭の docstring。

## 根拠
#34 / #65（ガント予実オーバーレイで配色を確定）、#35（CUD検証・調整の継続トラッカー・OPEN）。[`CLAUDE.md`](../../CLAUDE.md)「表示」「配色（CUD配慮・#35）」。
