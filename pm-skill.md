# /pm — 管理資料（Issue × WBS）メンテ

会話/指示の**文脈からモードを判断**し、GitHub Issue と WBS（roadmap json）を**一貫して維持**する。
判断は人間・進行と記録はこのスキル。$ARGUMENTS

## 真実の分担（重要）
- **Issue ＝ 問題・課題・完了条件**（WHAT / WHY / DONE）
- **WBS ＝ それに必要な作業の分解**（HOW / WHEN：工数・担当・日付・進捗）
- 繋ぐ鍵 ＝ **`#N`**（issue番号）。**1 issue → N 作業項目**。WBSの親ノード＝issueタイトル、子＝作業分解。
- 両方"真実"だが**持つ情報が違う**ので重複しない。**Issueに作業分解を、WBSに問題文を書かない。**

## プロジェクト設定
対象 repo ／ roadmap json のパス ／ version 方式 ／ issueタイトル規約 ／ **自動検証の実行コマンド** は、
**そのリポの CLAUDE.md（と非公開ルール .claude/rules）から読む**。無ければ人間に訊く。

## モード（AIが文脈で選ぶ）

### 1. 起票（「issue化して」／壁打ち後の合意）
1. **壁打ち**（進行フレーム・判断は人間）。少なくとも次を確認：
   - これは issue にする価値があるか？（些細なら起票しない＝過剰官僚化を防ぐ）
   - **高度 × 成果物**：この repo の対象か／別の高度 or 別成果物（＝別ツール）か
   - どの version / フェーズに置くか
2. 人間が「**起票して**」と言ったら確定。
3. **dedup**：`gh issue list --state all --search "<keywords> in:title,body"` で重複確認。あれば既存に追記。
4. **Issue作成**（構造）：背景 ／ **問題・課題** ／ 提案 ／ 論点 ／ **完了条件（受け入れ条件＝チェックボックス）** ／ 優先度 ／ 関連(#)。
5. **WBS反映**：roadmap の該当フェーズに、**issueタイトルを親ノード**として追加し、**必要な作業を子に分解**（plan日付・担当・工数）。親ノートに `#N：URL`。

### 2. 着手（「着手した」「やり始めた」）
- 対象 WBS リーフの `actual.start` に当日。必要なら issue に進行中の印。

### 3. 完了（「完了」「終わった」「close して」）
- **自動検証（"直したはず"で閉じない）**：
  - 完了条件のうち**機械で確認できるもの**（e2e／特定挙動／PIIグレップ 等）を**実際に実行**し、結果を提示。
  - **人間の目が要るもの**（見た目・UX・実機スモーク）は**人間確認を求める**。
- 全部満たしたら：**issue を close**（検証結果を要約コメント）＋ WBS リーフの `actual.end` に当日。

### 4. エラー自動起票（テスト/CI 失敗を検知）
- **dedup**（エラー署名で既存 open を検索）＋**閾値**（同一失敗が N 回連続で初めて起票・flaky を弾く）→ 自動で issue（再現手順／ログ／期待結果）＋ WBS 反映。
- ※ クライアント完結ツール（テレメトリ無し）は**テスト失敗のみ**が対象。運用エラーはサーバー側 repo で。

## ガードレール（毎回・例外なし）
- **push 前 PII grep**（ユーザー名・メール・絶対パス・実名・社内名）。
- コミットメール ＝ **GitHub noreply**。`git push --force` 禁止。
- サンプル/スクショのデータは**架空名のみ**（実在の同僚/顧客/案件名を書かない）。
- 公開リポに置くスキルは**写し（スナップショット）扱い**＝リポ側で独自編集せず、generic 版（このファイル）から再コピー。
- **「直したはず」で報告/close しない**（必ず自動検証を通す）。

## 出力
- やったこと（作成 issue#・WBS の変更点）を簡潔に。
- **判断の理由を1行残す**（なぜこの version／なぜ別ツール 等）＝後で揺れない。
